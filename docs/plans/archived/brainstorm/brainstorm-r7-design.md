## 一句話定義

Mind Flow 是一個 ADHD 友善的桌面知識管理工具，讓用戶用樹狀結構自由記錄多條思緒，
在分心後能零成本接回 context，並透過 AI 定期壓縮/萃取知識到持久知識庫。

## Context 生命週期

```
  ACTIVE ───手動 Archive──► ARCHIVED ───1天後自動──► VAULT
    ▲                          ▲                      │
    │                          │                      │
    └──────────────────────────┴──── 隨時手動喚回 ─────┘
```

- **ACTIVE**：正在使用，超過 1h（可調）未操作顯示灰色
- **ARCHIVED**：手動歸檔，1 天後自動進入 VAULT
- **VAULT**：封存，隱藏但可搜尋/喚回
- 不自動改狀態、不彈 toast 打擾（灰色顯示除外）

## AI 功能

### Summary（摘要）
- **觸發**：手動（快捷鍵 / context 右鍵選單）
- **行為**：分析當前 context tree 內容，彈出摘要視窗
- **不修改**原始 context 內容

### Compact（壓縮/合併）
- **觸發**：每日開啟 app 時自動 + 手動觸發（可在設定關閉自動）
- **兩種場景**：
  1. Active contexts 之間 → 建議合併相關/重複的 context
  2. Archived/Vault context → Common Knowledge → 建議萃取持久知識
- **確認 UI**：PR review 風格，AI 列出建議清單，用戶逐項勾選接受或拒絕

```
  ┌─ Compact Suggestions ──────────────────────────┐
  │                                                 │
  │  □ 合併「前端重構」和「CSS 優化」為一棵 tree     │
  │    理由：80% 節點內容重疊                        │
  │                                                 │
  │  □ 萃取「Auth系統/OAuth 筆記」到 Common Knowledge│
  │    理由：通用知識，不限於當前專案                  │
  │                                                 │
  │  □ 精簡「CI/CD」移除 3 個過時節點                │
  │    理由：內容已被更新的節點取代                    │
  │                                                 │
  │            [Apply Selected]  [Dismiss]           │
  └─────────────────────────────────────────────────┘
```

### Compact Review UI 方案

核心難點：tree 結構變更不像文字 diff 那麼直觀。我想了 3 種方案：

**方案 A：逐項變更卡片 + 迷你 tree diff（推薦）**

每個 AI 建議是一張卡片，卡片內用迷你 tree 圖展示 before/after：

```
┌─ Compact Review ──────────────────────────────────────────┐
│                                                           │
│  ┌─ 建議 1: 合併重複節點 ─────────── [Accept] [Skip] ──┐  │
│  │                                                      │  │
│  │  BEFORE                      AFTER                   │  │
│  │  前端重構                    前端重構                  │  │
│  │  ├── [T] CSS 優化            ├── [T] CSS 優化+筆記   │  │
│  │  ├── [T] CSS 筆記  ──合併►  │   (內容已合併)         │  │
│  │  └── [M] 重構計畫            └── [M] 重構計畫         │  │
│  │                                                      │  │
│  │  🔴 刪除: CSS 筆記                                    │  │
│  │  🔄 合併到: CSS 優化                                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                           │
│  ┌─ 建議 2: 萃取到 Common Knowledge ── [Accept] [Skip] ─┐  │
│  │                                                      │  │
│  │  FROM                        TO                      │  │
│  │  Auth系統                    Common Knowledge        │  │
│  │  └── [M] OAuth 筆記 ──複製►  └── [OAuth知識]         │  │
│  │                                                      │  │
│  │  📤 複製: OAuth 筆記 → Common Knowledge               │  │
│  │  ℹ️  原節點保留不動                                    │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                           │
│  ┌─ 建議 3: 移除過時節點 ─────────── [Accept] [Skip] ──┐  │
│  │                                                      │  │
│  │  CI/CD                                               │  │
│  │  ├── [T] Jenkins 設定    🔴 刪除（被 GitHub Actions 取代）│
│  │  ├── [T] 舊 deploy 流程  🔴 刪除（已過時）            │  │
│  │  └── [T] 環境變數筆記    🔴 刪除（已整合到 README）    │  │
│  │                                                      │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                           │
│                  [Apply All Accepted]  [Cancel]            │
└───────────────────────────────────────────────────────────┘
```

- 每張卡片獨立 Accept/Skip
- 卡片內有 BEFORE/AFTER 迷你 tree 或操作清單
- 底部有 🔴刪除 🔄合併 📤萃取 的操作摘要，一眼看懂
- 最後統一 Apply

**方案 B：全畫面 tree overlay**

直接在主畫面的 tree 上用顏色標記變更：

```
┌─────────────────────────────────────────────────┐
│ [Compact Review Mode]    建議 1/3     [◄] [►]   │
├─────────────────────────────────────────────────┤
│                                                 │
│              ┌── [T] CSS 優化+筆記  🟢 合併後   │
│              │                                  │
│  前端重構 ───┼── [T] CSS 筆記       🔴 將刪除   │
│              │                                  │
│              └── [M] 重構計畫       ⚪ 不變     │
│                                                 │
│  ┌──────────────────────────────────┐           │
│  │ 🔴 CSS 筆記 合併到 CSS 優化      │           │
│  │        [Accept]  [Skip]  [►Next] │           │
│  └──────────────────────────────────┘           │
└─────────────────────────────────────────────────┘
```

- 直接在 tree 上標色，最直觀
- 一次只看一個建議，用 ◄ ► 切換
- 缺點：一次只能看一個建議，整體感不夠

**方案 C：操作清單（最簡單）**

純文字列表，不畫 tree：

```
  ┌─ Compact Suggestions ──────────────────────────┐
  │                                                 │
  │  □ 合併 [前端重構/CSS 筆記] → [前端重構/CSS 優化]│
  │    🔴 刪除: CSS 筆記                             │
  │    🔄 內容合併到: CSS 優化                        │
  │                                                 │
  │  □ 複製 [Auth系統/OAuth 筆記] → Common Knowledge │
  │    📤 新建: OAuth知識 (Common Knowledge)          │
  │    ℹ️  原節點保留                                 │
  │                                                 │
  │  □ 刪除 [CI/CD] 下 3 個過時節點                  │
  │    🔴 Jenkins 設定、舊 deploy 流程、環境變數筆記   │
  │                                                 │
  │            [Apply Selected]  [Dismiss]           │
  └─────────────────────────────────────────────────┘
```

- 最簡單，開發成本最低
- 缺點：看不到 tree 結構變化的全貌

---

**我推薦方案 A**，理由：
1. 卡片式每個建議獨立，好 review
2. 迷你 tree diff 讓你看到結構變化但不佔太多空間
3. 底部操作摘要（🔴🔄📤）一眼掃到影響範圍
4. 開發難度介於 B 和 C 之間，UX 最好
