# PM-4: Tree 插入定位策略

> 產出日期：2026-02-23
> 角色：PM-4（Tree 插入定位）
> 前提：假設系統已確定 idea 要進入哪個 context（由上游分類模組決定）

---

## 核心理念

**「插在語意最近的地方，而非最方便的地方」**

ADHD 使用者的最大痛點之一是「事後整理」——丟進去之後找不到，或者結構越來越亂。因此插入定位策略的設計目標是：

1. **語意優先**：新 idea 應該靠近語意相近的節點，不是單純貼到 root 尾端
2. **零阻力**：不打斷使用者正在做的事，不要求使用者手動決定位置
3. **可修正**：使用者看到結果後能快速調整，不需要找到原始捕捉入口
4. **可解釋**：系統要能告訴使用者「為什麼放在這裡」，建立信任感

---

## 定位演算法描述

### 階段一：取得樹的語意快照

從後端 `get_tree(context_id)` 取得完整 `TreeData`，對每個節點建立**語意描述字串**：

```
"{title} [父節點:{parent.title}] [兄弟:{sibling_titles_joined}]"
```

這個字串作為語意比較的基準單元，不需要深度語意模型，可用輕量字串相似度（TF-IDF 向量或 embedding）。

### 階段二：候選父節點評分

對所有「非葉子」節點（depth < MAX_DEPTH - 1）和根節點計算**親和分數（affinity score）**：

| 評分維度 | 權重 | 說明 |
|---------|------|------|
| 標題語意相似度 | 40% | idea 文字 vs 節點標題的 cosine similarity（本地 embedding 或字元 n-gram） |
| 路徑語意相似度 | 20% | idea 文字 vs 從根到該節點的完整路徑 |
| 時間接近度 | 20% | 越近期更新的節點略有加分（ADHD 使用者傾向在同一個思路脈絡中持續發展） |
| 深度懲罰 | -10% | 越深的節點扣分（避免過度巢狀，ADHD 友善的扁平化偏好） |
| 子節點飽和懲罰 | -10% | 子節點數超過 6 的節點扣分（避免某個節點無限膨脹） |

取**前 3 名**候選節點，最高分者為預設插入目標。

### 階段三：插入位置決定

確定父節點後，決定在兄弟中的**排序位置**：

- **語意接近的兄弟旁邊**：與已有兄弟節點做相似度比較，插在最相近的兄弟節點之後
- **若沒有相近兄弟**：插在該父節點的末端（position = max + 1）
- **特殊情況**：若最高親和分數 < 閾值（例如 0.25），直接掛在 root 下末端，並標注「未找到合適位置」

### 階段四：是否自動建立中間節點

若 idea 文字包含明確的分類信號（例如「關於 X 的…」「X 的 Y 想法」），且現有樹中沒有 X 對應的節點，考慮：

- 分數差距懸殊（最高分與第二名差 > 0.3）→ 直接插入，不建立中間節點
- 分數集中（前三名都差不多）→ 可建議建立新的「X」分類節點作為中間層

**預設不自動建立中間節點**（避免 ADHD 使用者面對突然冒出來的新分類感到困惑），改為「建議建立」，讓使用者一鍵確認。

## 插入位置決策流程圖（文字）

```
[收到 idea + target context_id]
         │
         ▼
[get_tree 取得完整樹結構]
         │
         ▼
[對所有可選父節點計算 affinity score]
         │
         ├─ 最高分 >= 閾值(0.25) ──→ [選最高分節點為父節點]
         │                                      │
         └─ 最高分 < 閾值 ──────────→ [改用 root 作為父節點]
                                               │
                              ┌────────────────┘
                              ▼
                 [在父節點的子節點中找最相近的兄弟]
                              │
                 ├─ 有相近兄弟 ──→ [插在該兄弟之後]
                 └─ 無相近兄弟 ──→ [插在末端]
                              │
                              ▼
              [呼叫 ipc.createNode(context_id, parent_id, type, title)]
                              │
                              ▼
              [回傳新節點 ID + 插入理由說明字串]
```

## 視覺回饋設計

### 新節點高亮（Capture Highlight）

與 AI Compact 的 highlight 系統共用 `compactHighlights` map 的**顏色 slot 機制**，但使用不同的顏色語意：

| 狀態 | 色帶 | 背景 | 標籤 |
|------|------|------|------|
| 剛插入 | #A78BFA99（紫色） | #2D1F3A | `NEW` 小 badge |
| 插入 5 秒後 | 漸出（transition 700ms） | 無 | badge 消失 |

### 路徑高亮

插入後，自動展開從 root 到新節點的完整路徑（折疊狀態的節點自動展開），讓使用者立即看到新節點在哪裡。

### ScrollIntoView

呼叫 `scrollIntoView({ behavior: "smooth", block: "nearest", inline: "nearest" })` 把新節點捲入視窗，同時 `selectNode(newNodeId)` 讓新節點被選中。

### 插入理由 Toast / Banner

在 StatusBar 下方（CompactBanner 的位置）短暫顯示 1 行說明：

```
[紫色點] idea 已加到「{父節點標題}」下 — 因為語意最相近
```

- 3 秒後自動消失
- 點擊可跳到該節點
- 不使用 CompactBanner（那是 AI Compact 專用，避免視覺混淆）

### 低信心提示

當最高 affinity score < 0.25（掛到 root 下）：

```
[橘色點] idea 暫時放在樹根 — 沒有找到合適的位置，可手動移動
```

## 使用者修正流程

### 即時修正（推薦路徑）

插入後新節點被選中，使用者可立即用現有的鍵盤快捷鍵調整位置：

| 動作 | 快捷鍵 |
|------|--------|
| 上移同級 | Alt+↑ |
| 下移同級 | Alt+↓ |
| 移入前一兄弟節點 | Alt+→ |
| 移出到祖父節點 | Alt+← |
| 拖放到任意位置 | 拖放 |

這些都是現有功能，不需要額外開發。

### 延遲修正

若使用者沒有立即調整，未來可透過以下方式找到並修正：

1. **Cmd+F 搜尋節點**：用關鍵字找到捕捉的 idea
2. **AI Compact**：日後在該 context 執行 AI Compact，自動重組整棵樹（包含新插入的節點）

### 「撤回捕捉」操作

在插入後的 Toast Banner 上提供「撤回」按鈕（3 秒內），點擊後刪除剛建立的節點（呼叫 `ipc.deleteNode`），讓使用者重來。這個操作也會加入 undo stack（undoEntry type: `add`），所以 Cmd+Z 也能撤回。

## 與 AI Compact 的搭配

### 互補關係

| 功能 | AI Compact | 插入定位 |
|------|-----------|---------|
| 觸發時機 | 使用者主動要求 | 自動（背景） |
| 作用範圍 | 選定子樹的內部重組 | 單一節點的位置決定 |
| 結果 | 整棵子樹被重新組織 | 一個新節點出現在某處 |
| Undo | 整棵子樹復原 | 單一節點刪除 |

### 協作流程

```
快速捕捉 idea（自動插入） → 累積若干節點
         ↓
選擇某個 context 執行 AI Compact
         ↓
AI Compact 同時整理原有節點 + 最近捕捉進來的 idea 節點
         ↓
使用者確認 / 修改 Compact 結果
```

這樣使用者可以「先捕捉、後整理」，符合 ADHD 思維節奏。

### 注意事項：Compact 鎖定期間

若目前 context 的 `compactState === APPLIED`（Compact 鎖定中），插入定位不應往 compact 子樹外部插入節點（與現有的 `isCompactLocked` 檢查一致）。這種情況下有兩個選項：

1. **選項 A（保守）**：拒絕插入，Toast 提示「請先確認或復原 AI Compact 再捕捉」
2. **選項 B（激進）**：改插到 compact 子樹內部的合適位置

建議採用**選項 A**，行為可預期、不干擾 Compact 工作流。

---

## 優點

1. **零摩擦捕捉**：使用者不需要切換 context、不需要決定位置，直接說出 idea 就好
2. **語意自組織**：隨著使用，樹的語意密度提高，後續插入精準度也會提升（因為有更多節點可以參照）
3. **與現有架構兼容**：完全利用現有 `create_node` IPC、`selectNode` store action、`scrollIntoView` 機制，不需要大幅改造後端
4. **ADHD 友善**：色彩高亮 + 5 秒自動消失，不留下視覺噪音；失敗降級（掛 root 下）有明確說明，不讓使用者困惑
5. **可調整性高**：閾值、權重都是可以透過 AI Profile 設定暴露給進階使用者的參數

## 風險與挑戰

### 技術風險

| 風險 | 說明 | 緩解策略 |
|------|------|---------|
| **語意計算在前端成本高** | 每次插入都要對全樹計算相似度，大樹（50+ 節點）可能有感延遲 | 後端新增 `find_best_parent(context_id, query)` Rust command，用 SQLite FTS5 或輕量向量計算 |
| **LLM 依賴** | 若使用 LLM 做語意比對，每次捕捉都要呼叫 API，成本高且離線失效 | 優先使用本地計算（n-gram 相似度），LLM 只用於分類 context，不用於定位 |
| **Compact 鎖定衝突** | 捕捉時剛好在 Compact APPLIED 狀態 | 如上所述採選項 A（拒絕插入 + 提示） |
| **樹結構變更的競態** | 使用者正在拖放節點，同時插入新節點，可能造成 position 衝突 | 樂觀插入到末端（position = max + 1），衝突機率極低；`move_node` 有 shift 邏輯可處理 |

### 使用者體驗風險

| 風險 | 說明 | 緩解策略 |
|------|------|---------|
| **插錯位置卻沒注意到** | 使用者不看 Toast，idea 默默插在奇怪的地方 | 高亮持續 5 秒 + ScrollIntoView 確保使用者看到；Cmd+F 可隨時搜尋 |
| **語意錯誤積累** | 初期樹稀疏，定位精準度低，錯誤插入的節點反過來影響後續判斷 | 閾值機制（低分掛 root）避免誤導；AI Compact 可週期性修正整棵樹 |
| **中間節點建議打斷心流** | 若系統頻繁提示「要建立新分類嗎？」會讓 ADHD 使用者分心 | 預設關閉中間節點建議，作為進階選項 |
| **與 CompactBanner 視覺混淆** | 兩種 banner 同時可能存在（雖然 Compact 鎖定期間不允許插入） | 使用不同顏色（紫色 vs 橘色）和不同位置，並在 Compact APPLIED 期間禁止捕捉 |
