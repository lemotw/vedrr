# PM-5: 跨 Context 與新 Context 策略

> 產出日期：2026-02-23
> 角色：PM-5（跨 Context 與邊界情況）

---

## 核心理念

「隨手丟 idea，系統智慧分入」的功能不只是簡單的一對一分類。現實中，使用者的思維往往跨越邊界：一個 idea 可能同時屬於多個領域，可能完全超出現有框架，也可能是多個散落碎片的聚合點。本方案的核心信念是：

- **模糊性是正常的**：強迫每個 idea 只屬於一個 context 是對思維的暴力切割
- **使用者判斷優先**：AI 是建議者，不是決策者；分類錯誤必須容易修正
- **漸進式確定性**：允許 idea 從「未分類」→「候選」→「已歸屬」的自然流動
- **隱私是前提**：部分 idea 不應被 AI 讀取，這是系統設計而非例外處理

## 多 Context 關聯方案

### 問題場景

使用者輸入：「用 Rust 寫一個 CLI 工具來管理我的閱讀清單」

這個 idea 可能同時屬於：
- `Rust 學習` context（技術實踐）
- `閱讀習慣` context（工具輔助）
- `Side Projects` context（專案規劃）

### 方案 A：主歸屬 + 鏡像引用（推薦）

**概念**：一個 idea 有一個「主 context」，其他相關 context 持有「引用節點」

```
主 context: Side Projects
  └── [實際節點] 用 Rust 寫閱讀清單 CLI

關聯 context: Rust 學習
  └── [引用節點] → 指向 Side Projects 的節點

關聯 context: 閱讀習慣
  └── [引用節點] → 指向 Side Projects 的節點
```

**實作方向**：
- DB 加入 `node_references` 表（`source_node_id`, `target_context_id`, `target_node_id`）
- 引用節點在 UI 上用特殊樣式標示（如斜體、虛線邊框、不同 badge）
- 引用節點不可編輯，點擊可跳轉到原節點所在 context
- 刪除原節點時，引用節點自動變為「孤兒引用」（可選擇保留或轉為實體）

**優點**：避免資料重複；修改一處自動反映所有引用；清楚區分原始 vs 引用

**缺點**：實作複雜度高（需要新 DB 表 + UI 跳轉機制）；跨 context 的引用關係難以視覺化

### 方案 B：複製並標記（簡單版）

**概念**：複製到多個 context，但附加 `source_context_id` 元資料，標示原始來源

- 使用現有 `clone_subtree` 機制
- 在節點 `content` 欄位用隱藏格式記錄 `linked_from: context_id/node_id`
- UI 顯示「來自 X context」的提示，點擊可跳轉

**優點**：實作簡單；不需新 DB 表；Undo 語義清楚

**缺點**：資料重複；原節點修改後複本不同步；長期形成資訊孤島

### 方案 C：標籤交叉（Tag 橋接）

不複製節點，改用 tag 讓節點在不同視角下可見（見 Tag vs Context 分析章節）

### 決策建議

**短期**：採用方案 B（複製 + 標記），實作快速，可先驗證使用者行為。
**中期**：根據使用者回饋決定是否升級到方案 A（引用機制）。
**長期**：方案 A + Tag 系統配合，形成完整的知識圖譜基礎。

## 新 Context 自動建立機制

### 觸發條件

AI 建議建立新 context 的條件（需同時滿足）：

1. **語義距離**：新 idea 與所有現有 context 的語義相似度低於閾值（< 0.3）
2. **孤立性**：無法作為任何現有 context 的節點（無合理父節點）
3. **完整性**：idea 本身具有足夠的獨立性（能成為樹的根節點）

### 流程設計

```
使用者輸入 idea
       ↓
AI 分析：與現有 context 的相關性評分
       ↓
    ┌──────────────────────────────┐
    │ 所有評分 < 0.3               │
    └──────────┬───────────────────┘
               ↓
    提示使用者：「這個 idea 不太屬於現有任何 context。
    建議建立新 context：[AI 建議名稱]
    或選擇現有 context：[列表]」
               ↓
    ┌─────────────────────┐
    │ 使用者確認           │ → 自動建立新 context（使用建議名稱）
    │ 使用者修改名稱       │ → 建立（自訂名稱）
    │ 使用者選擇現有       │ → 歸入現有 context
    │ 使用者取消           │ → 放入收集箱
    └─────────────────────┘
```

### 名稱建議策略

AI 建議 context 名稱時的原則：
- 名稱應為「領域 / 主題」形式，而非描述性句子
- 長度限制：2-8 個中文字或 3-20 個英文字元
- 提供 2-3 個候選名稱供使用者選擇（不只一個）

**例子**：idea「研究 Obsidian 的插件系統架構」
- 候選 1：「工具研究」
- 候選 2：「PKM 工具」
- 候選 3：「Obsidian 探索」

### 新 Context 的初始結構

自動建立的 context 不應只有根節點，AI 可以：
1. 將觸發此次建立的 idea 作為第一個子節點
2. 根據 idea 內容建議 2-3 個可能的子分類（虛線框，等待使用者確認）

## Tag vs Context 分析

### 核心差異

| 維度 | Context | Tag |
|------|---------|-----|
| 結構 | 樹狀（有層次） | 扁平（無層次） |
| 內容 | 一棵完整的思維樹 | 一個屬性標籤 |
| 適合 | 深度探索某個主題 | 橫向連結多個主題 |
| 切換成本 | 高（需要 Cmd+K） | 低（即時篩選） |
| AI 分類 | 語義相似性 | 關鍵字/概念匹配 |

### Mind Flow 的定位

Mind Flow 以 context 為核心，已有穩固的樹狀思維模型。Tag 系統是補充，不是替代。

**建議的 Tag 定位**：
- Tag 是跨 context 的「視角」，讓使用者能以不同維度檢視散落各處的節點
- 不引入複雜的 tag hierarchy（避免增加認知負擔）
- Tag 不影響 context 內的樹結構

**需要 Tag 的場景**：
- 「#待辦」：跨所有 context 的行動項目
- 「#靈感」：各 context 中的創意閃現
- 「#參考資料」：需要引用的外部資源
- 「#問題」：尚未解答的問題

**不需要 Tag 的場景**：
- 主題分類（這是 context 的職責）
- 優先順序管理（用節點位置表達）
- 節點類型（已有 TEXT/MARKDOWN/IMAGE/FILE 四種）

### 實作建議

Tag 系統已存在於 DB schema（`contexts.tags` 欄位為 JSON array），目前未在 UI 中使用。

短期可先在 node 層級引入 tag（需新增 `node_tags` 欄位或表），搭配 idea 收集功能一起推出：
- idea 分類時可附加 tag
- 之後可透過 tag 跨 context 搜尋

## Idea 聚合邏輯

### 場景：散落 ideas 自動聚合成新 context

假設使用者的收集箱（Inbox context）累積了 30 個 ideas，其中有以下群集：

```
idea-1: 學 SwiftUI 的 animation API
idea-7: Metal shader 的效能調校
idea-15: macOS 的 Core Data vs SQLite 比較
idea-23: Xcode 16 的新功能測試
```

這 4 個 ideas 都屬於「Apple 平台開發」，AI 應該建議聚合。

### 聚合流程

```
觸發時機：
  1. 收集箱達到一定數量（如 20 個）
  2. 使用者手動觸發（「整理 Inbox」按鈕）
  3. 定期提示（如每週一次）

AI 分析流程：
  1. 計算所有 ideas 之間的語義相似度矩陣
  2. 聚類演算法（K-means 或 DBSCAN）找出自然群集
  3. 為每個群集建議 context 名稱
  4. 呈現聚合預覽（類似 AI Compact 的 banner 模式）

使用者確認介面：
  顯示建議的 context 群集，每個群集可以：
  - 確認建立新 context
  - 合併到現有 context
  - 拆分此群集
  - 保留在 Inbox（不聚合）
```

### 聚合後的樹結構

聚合後的新 context，其樹結構不應是扁平的 ideas 列表，AI 應嘗試建立合理的層次：

```
Apple 平台開發（root）
  ├── iOS / macOS 開發
  │    ├── SwiftUI animation API
  │    └── Core Data vs SQLite 比較
  ├── 效能與底層
  │    └── Metal shader 效能調校
  └── 工具與生態
       └── Xcode 16 新功能
```

## 隱私考量

### 問題核心

不是所有 idea 都適合讓 AI 讀取。典型的隱私敏感 idea：
- 個人健康狀況、心理狀態
- 財務計畫、薪資資訊
- 對特定人物的評價
- 商業機密、未公開計畫

### 設計原則

**隱私是使用者的權利，不應是需要特別申請的功能。**

### 隱私保護機制

#### 節點層級：私密標記

每個節點可標記為「私密」：

- 私密節點在 AI 分析時完全排除（不納入 prompt）
- 私密節點在 UI 上用特殊樣式顯示（如鎖頭圖示）
- 私密節點不出現在 AI Compact 的重組範圍內
- 快捷鍵：`p` 切換私密狀態（類似 `z` 切換摺疊）

#### Context 層級：私密 Context

整個 context 可標記為「私密」：

- 私密 context 不被納入任何 AI 分析
- 不出現在 AI 的 idea 分類建議中
- QuickSwitcher 仍顯示，但有鎖頭標記
- 已有 `vault` 狀態（contexts.state），可擴展其語義

#### 本地優先的隱私保障

Mind Flow 本身已是本地優先（無雲端同步），這提供了基礎隱私保障：

- AI 分析在本地完成（未來可考慮本地 LLM，如 Ollama）
- 使用雲端 LLM 時，只發送使用者明確允許的內容
- 明確的資料流說明：「以下內容將傳送至 [Provider]：...」

#### 實作優先順序

1. **先做**：節點層級的私密標記（DB 加 `is_private` 欄位）+ AI 分析時排除
2. **中期**：Context 層級的私密標記（擴展 `vault` 語義）
3. **長期**：本地 LLM 選項（Ollama 整合）

## 優點

1. **符合真實思維模型**：人類思維天然跨界，系統設計尊重這一事實而非強迫分類
2. **漸進承諾**：使用者不必在 idea 輸入時就做最終決定，可以先「丟進去」再整理
3. **AI 輔助但不主導**：每個 AI 建議都有確認步驟，避免自動分類造成的不信任
4. **隱私先行**：私密機制是內建功能，不是事後補丁
5. **與現有架構相容**：主要在現有 context/node 系統上疊加，不需根本性重構
6. **可測試的價值假設**：聚合功能本身就是對「使用者真的會累積 inbox」假設的驗證

## 風險與挑戰

### 技術風險

| 風險 | 可能性 | 影響 | 緩解方案 |
|------|--------|------|---------|
| LLM 語義相似度計算成本高 | 高 | 中 | 使用本地 embedding 模型（sentence-transformers）或降低計算頻率 |
| 多 context 引用的 DB 查詢複雜 | 中 | 中 | 分階段引入，先複製後引用 |
| 聚合算法在小樣本下不穩定 | 高 | 低 | 設定最低樣本門檻（至少 5 個 ideas 才建議聚合） |
| 引用節點的 undo 語義複雜 | 中 | 中 | 引用節點的建立/刪除獨立於主 undo stack |

### UX 風險

| 風險 | 描述 | 緩解方案 |
|------|------|---------|
| 分類建議品質低 → 使用者不信任 | AI 分錯一次，使用者可能永遠不再使用自動分類 | 提供清楚的「為什麼這樣分類」說明 + 一鍵修正 |
| 確認步驟過多 → 使用者放棄 | 每次丟 idea 都要確認，比直接分類還麻煩 | 預設行為：先丟入 Inbox，批量確認而非逐一確認 |
| 私密標記被遺忘 | 忘記標記的敏感 idea 被 AI 分析 | 引入「敏感關鍵字提示」（偵測到敏感詞時主動提示） |
| 新 Context 爆炸 | AI 過度建議建立新 context，使用者眼花撩亂 | 限制 AI 建議頻率；Context 數量達上限時主動建議合併 |

### 認知負擔風險

「跨 Context 引用」「Tag 系統」「聚合機制」同時存在，可能讓 Mind Flow 從「簡單」變成「複雜」。

**緩解原則**：
- 功能分層：基本功能（一個 idea → 一個 context）不受影響
- 進階功能（引用、聚合）是使用者主動觸發的，不強迫使用
- ADHD 友善設計始終優先：系統應該幫使用者減少決策，不是增加

## 實作優先順序建議

| 階段 | 功能 | 複雜度 |
|------|------|--------|
| Phase 1 | 收集箱（Inbox context）+ 一鍵分類到現有 context | S |
| Phase 1 | 節點私密標記（DB + UI + AI 排除） | S |
| Phase 2 | 完全不屬於任何 context 時，建議新 Context | M |
| Phase 2 | 多 context 相關時，建議複製到多個（方案 B） | M |
| Phase 3 | Inbox 批量聚合建議 | L |
| Phase 4 | 跨 context 節點引用機制（方案 A） | XL |
| Phase 4 | Tag 系統（節點層級） | L |
